## 6. 风险与边界

### 6.1 风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| subagent token 消耗大（双盲 2× 审查） | 高 | 中 | 控制 A、B 提示词 ≤ 800 字，结果只输出关键问题不过度展开 |
| v12 新代码与旧问题混淆 | 中 | 中 | 在 `audit-merged.md` 明确标"v12-新增"vs"长期遗留" |
| main 分支 vs feature 分支不一致 | 低 | 低 | 审查基于工作树快照（已与 main 同步） |
| 文档同步可能漏 | 中 | 低 | 用 TodoWrite 列 6 份文档，每份独立确认 |
| subagent 输出格式不一致 | 中 | 中 | 我在 Step 3 统一格式与分级 |

- ❌ 不重做已完成的 v12 功能
- ❌ 不在 main 分支上修改（用 `feature/v13-code-audit` 新分支）

## 7. 验收标准

| 维度 | 验收标准 |
|------|----------|
| **报告完整性** | `audit-merged.md` 覆盖 6 维，Top10 + P0/P1/P2/P3 全部分级，每条带文件:行号 |
| **互盲验证** | subagent A 与 B 都能给出 ≥ 5 条独立问题（去重前） |
| **仲裁质量** | 共识/独报/仲裁三类标签清晰；冲突项有书面理由 |
| **文件交付** | design spec + 实施计划都通过自审（无 TBD/矛盾/歧义） |
| **文档同步** | 6 份项目文档同步更新，commit 描述清楚"v13 起点"含义 |
| **工程纪律** | 全程在 `feature/v13-code-audit` 新分支进行；只在文末 1 个 commit |
| **Token 经济** | subagent A + B 总产出 ≤ 50KB markdown（控制噪声） |
